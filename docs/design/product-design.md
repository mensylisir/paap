# PAAP 平台产品设计方案

## 一、产品定位

**PAAP (Platform-as-a-Application Platform)** — 以应用为中心的自助式云原生应用管理平台。

核心转变：从"管理员分配资源给租户"转变为"应用管理员自助管理应用全生命周期"。

---

### 3.4 环境管理

**功能：** 管理应用的环境。每个环境是一个独立的交互式画布，用户可以在其中安装工具、创建组件。

**设计要点：**
- 环境可以是空的，用户按需装工具、建组件
- 每个环境拥有独立的 ArgoCD/Jenkins/Prometheus 实例（不是共享的）
- 不同环境的工具链可以不同（如开发环境有CI+CD，生产环境只有CD）
- 创建新环境时，根据模板安装对应的工具实例（或空环境按需安装）

```
│  订单服务 - 环境                    [+ 创建环境]     │
```

## 四、服务抽象层设计

### 4.1 环境内资源分类

每个环境内有三类资源，用户可以在环境画布中分别管理：

**工具（管理类服务）** — 帮助用户管理部署、构建、监控等流程

| 用户可见 | 底层技术 | 说明 |
|----------|----------|------|
| 部署服务 | ArgoCD | 管理应用部署、版本、回滚 |
| CI服务 | Tekton/Jenkins | 管理构建流水线 |
| 监控服务 | Prometheus/Grafana | 资源监控与告警 |
| 日志服务 | EFK/Loki | 日志收集与查询 |

**组件（业务服务）** — 用户自己的业务应用

| 类型 | 说明 | 示例 |
|------|------|------|
| 前端服务 | Web 应用、静态资源 | Vue/React 应用 |
| 后端服务 | API 服务、微服务 | Spring Boot/Go 服务 |
| 自定义 | 用户自定义的任意服务 | 网关、BFF 等 |

**基础设施服务** — 为组件提供运行所需的底层能力

| 用户可见 | 底层技术 | 说明 |
|----------|----------|------|
| 数据库 | PostgreSQL/MySQL/人大金仓 | 关系型数据库 |
| 缓存 | Redis/Memcached | 缓存服务 |
| 消息队列 | RabbitMQ/Kafka/RocketMQ | 异步消息 |
| 对象存储 | MinIO | 文件存储 |
| 注册中心 | Nacos/Eureka | 服务注册发现 |
| 配置中心 | Nacos/Apollo | 配置管理 |

### 4.2 服务生命周期

```
┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
│ 申请   │ →  │ 配置   │ →  │ 使用   │ →  │ 管理   │
│ 服务   │    │ 服务   │    │ 服务   │    │ 服务   │
└────────┘    └────────┘    └────────┘    └────────┘
  选择服务      填写表单      查看状态      扩缩容
  选择环境      勾选选项      查看监控      更新配置
  点击确认      点击保存      查看日志      停用服务
```

### 4.3 服务配置原则

1. **表单优先：** 所有配置项以表单形式呈现，支持勾选、下拉、输入
2. **合理默认：** 每个配置项都有合理的默认值，用户只需修改差异部分
3. **渐进暴露：** 基础配置默认展示，高级配置折叠隐藏
4. **禁止YAML：** 不向用户暴露任何 YAML/JSON 配置文件

## 五、环境与模板机制

### 5.1 环境模型

环境名由用户自定义，不限于 dev/staging/prod。每个环境是一个独立的运行单元，**包含一组 namespace 和一套独立的工具实例**。

**名称与标识：** 所有资源都有名称（展示用，可中文）和标识（K8s 资源用，纯英文）。例如：应用名称「订单服务」→ 标识 `order-service`，环境名称「开发环境」→ 标识 `dev`，工具名称「部署服务」→ 标识 `argocd`。

```
应用「订单服务」（标识: order-service）
 │
 ├── 环境: "开发环境"（标识: dev）
 │    ├── 主空间: order-service-dev              ← 工具 SA 放这里
 │    ├── 工作负载空间: order-service-dev-app     ← ArgoCD 部署组件
 │    ├── 专用空间: order-service-dev-db          ← 数据库
 │    ├── 专用空间: order-service-dev-cache       ← 缓存
 │    ├── 工具实例（每环境独立）：
 │    │    ├── ArgoCD（order-service-dev-argocd）     ← 覆盖全部 dev 域 namespace
 │    │    ├── Tekton（order-service-dev-tekton）     ← 覆盖全部 dev 域 namespace
 │    │    └── Prometheus（order-service-dev-prom）   ← 采集全部 dev 域 namespace
 │    ├── 组件: 前端服务 · 后端服务
 │    ├── 基础设施: PostgreSQL · Redis
 │    └── RBAC: 工具 SA 在 dev ns，通过 RoleBinding 获得所有 dev 域 namespace 权限
 │
 ├── 环境: "测试环境"（标识: staging）
 │    ├── 主空间: order-service-staging
 │    ├── 工作负载空间: order-service-staging-app
 │    ├── 工具实例（另一套，互不干扰）：
 │    │    ├── ArgoCD（order-service-staging-argocd）
 │    │    ├── Tekton（order-service-staging-tekton）
 │    │    └── Prometheus（order-service-staging-prom）
 │    ├── 组件: 前端服务 · 后端服务
 │    └── 基础设施: PostgreSQL · Redis
 │
 └── 环境: "生产环境"（标识: prod）
      ├── 主空间: order-service-prod
      ├── 工作负载空间: order-service-prod-app
      ├── 工具实例：
      │    ├── ArgoCD（order-service-prod-argocd）       ← 只有 CD，没有 CI
      │    └── Prometheus（order-service-prod-prom）
      └── 组件: 前端服务 · 后端服务

注意：不同环境的工具实例完全独立。
开发环境的 ArgoCD 碰不到测试环境的任何资源。
```

**关键设计决策：**
- 环境 ≠ namespace。一个环境 = {主 namespace + 可选附加 namespace} + {一套环境级工具实例}
- 环境名由用户自定义（开发环境、联调环境、UAT验收...），namespace 自动加前缀避免冲突
- 工具实例**按环境隔离**，不是按应用隔离。每个环境有自己的 ArgoCD/Tekton/Prometheus
- 工具 SA 统一放在环境的主 namespace，通过 RoleBinding 获取该环境所有 namespace 的权限
- 环境内 namespace 按用途分三类：主空间、工作负载空间、专用空间
- 环境新增 namespace 时，平台自动为已安装的工具创建新 RoleBinding
- 监控/日志默认采集该环境所有 namespace，权限不外溢到其他环境
- 环境内可以有三类资源：**工具**（ArgoCD/Jenkins/Prometheus）、**组件**（业务服务）、**基础设施服务**（数据库/缓存）
- 空环境 = 只创建了 namespace，没有安装任何工具
- 网络隔离默认使用 NetworkPolicy，IP Pool 为可选高级功能

### 5.2 环境模板结构

```yaml
# 模板定义（管理员维护，用户不可见YAML）
模板: 开发环境标准
描述: 适用于日常开发，包含完整的CI/CD和监控能力
包含服务:
  - 部署服务:
      默认副本数: 1
      自动部署: true
  - CI服务:
      默认触发: 代码推送
      缓存构建: true
  - 监控服务:
      采集间隔: 30s
      保留时长: 7天
资源配额:
  CPU: 4核
  内存: 8GB
  存储: 50GB
```

### 5.3 一键建环境流程

```
用户选择模板
    ↓
系统检查资源配额
    ↓
后台自动创建:
  - Kubernetes Namespace（环境主 namespace）
  - 根据模板安装工具实例（每环境独立一套，如 ArgoCD/Tekton/Prometheus）
  - 为每个工具创建 ServiceAccount + Role + RoleBinding
  - 网络策略
  - 资源配额
    ↓
```

## 七、权限模型

### 7.1 角色定义

| 角色 | 职责 | 可见菜单 |
|------|------|----------|
| 平台管理员 | 管理集群、模板、共享服务 | 全部 |
| 应用管理员 | 管理应用内所有资源 | 我的应用 + 应用级菜单 |
| 应用成员 | 查看应用状态，执行部署 | 我的应用 + 应用级菜单（只读+部署） |
| 访客 | 只读查看 | 我的应用（只读） |

### 7.2 权限隔离

- **应用级隔离：** 用户只能看到自己有权限的应用
- **环境级隔离：** 不同环境的资源相互独立
- **服务级隔离：** 通过 ServiceAccount 自动绑定，用户无需感知
- **工具实例隔离：** ArgoCD/Prometheus/Tekton 等工具的每个环境实例只能访问自己环境内的 namespace，不同环境的工具实例完全独立，详见 [服务实例权限隔离设计方案](service-isolation-design.md)

## 八、关键交互流程

### 8.1 应用管理员首次使用流程

```
登录平台
  ↓
看到「我的应用」（空状态）
  ↓
点击「创建应用」
  ↓
填写应用名称、标识、描述
  ↓
选择环境模板（如：快速开始）
  ↓
系统自动创建应用 + 3个环境（每个环境独立安装工具实例）
  - 开发环境: ArgoCD + Tekton + Prometheus（完整工具链）
  - 测试环境: ArgoCD + Tekton + Prometheus（完整工具链）
  - 生产环境: ArgoCD + Prometheus（只有CD和监控，无CI）
  ↓
进入应用概览，看到3个环境的运行状态
  ↓
点击「部署服务」→ 部署第一个版本
  ↓
点击「CI服务」→ 配置自动构建流水线
  ↓
点击「监控服务」→ 查看资源使用情况
```

### 8.2 日常运维流程

```
开发者推送代码
  ↓
CI流水线自动触发（用户无需操作）
  ↓
构建成功后自动部署到开发环境
  ↓
应用管理员收到通知
  ↓
在部署服务中查看部署状态
  ↓
确认无误后，手动触发部署到测试环境
  ↓
测试通过后，手动触发部署到生产环境
  ↓
在监控服务中观察生产环境运行状态
```

### 8.3 告警处理流程

```
监控服务检测到异常
  ↓
触发告警规则
  ↓
通知应用管理员（站内消息/邮件/企业微信）
  ↓
应用管理员查看监控详情
  ↓
定位问题组件
  ↓
查看组件日志
  ↓
决定处理方式:
  - 回滚到上一版本
  - 调整资源配置
  - 修复代码重新部署
```

## 九、界面规范

### 9.1 术语对照表

| 禁止使用的术语 | 应使用的术语 |
|---------------|-------------|
| Namespace | 空间（环境内的资源分组） |
| Deployment | 组件/服务 |
| Service | 服务访问入口 |
| Pod | 实例 |
| Container | 运行单元 |
| ServiceAccount | （不展示） |
| RBAC | （不展示） |
| YAML | （不展示） |
| Helm Chart | （不展示） |
| ArgoCD | 部署服务 |
| Jenkins/Tekton | CI服务 |
| Prometheus | 监控服务 |
| Loki/EFK | 日志服务 |
| Ingress | 访问地址 |
| ConfigMap/Secret | 配置项 |
| NetworkPolicy | 网络隔离（高级设置中展示） |
| IPPool | IP地址池（高级设置中展示） |

**命名规则：**
- 所有资源都有**名称**（展示用，可中文）和**标识**（K8s 用，纯英文）
- 应用名称「订单服务」→ 标识 `order-service`
- 环境名称「开发环境」→ 标识 `dev`
- 工具名称「部署服务」→ 标识 `argocd`
- Namespace 命名：`{应用标识}-{环境标识}`（主空间），`{应用标识}-{环境标识}-{用途}`（附加空间）

### 9.2 操作步骤限制

所有核心操作不超过4步：

| 操作 | 步骤 |
|------|------|
| 创建应用 | 填信息 → 选模板 → 确认 |
| 部署版本 | 选环境 → 选组件 → 选版本 → 确认 |
| 创建流水线 | 填名称 → 选仓库 → 选步骤 → 确认 |
| 配置监控 | 选方案 → 选通知方式 → 确认 |
| 创建环境 | 选模板 → 填名称 → 确认 |

### 9.3 状态展示规范

- 使用绿色/红色/灰色圆点表示状态，不使用技术性状态码
- 时间使用相对时间（"10分钟前"），不使用绝对时间戳
- 资源使用使用进度条/百分比，不使用原始数值
- 事件使用自然语言描述，不使用技术性日志

## 十、技术架构建议

### 10.1 前端技术栈

- **框架：** React 18 + TypeScript
- **UI库：** Ant Design 5.x / Arco Design
- **状态管理：** Zustand / React Query
- **路由：** React Router v6
- **构建：** Vite

### 10.2 后端技术栈

- **框架：** Go (Gin/Fiber) 或 Node.js (NestJS)
- **API：** RESTful + WebSocket（实时状态推送）
- **数据库：** PostgreSQL
- **缓存：** Redis
- **消息队列：** RabbitMQ（异步任务）

### 10.3 与底层平台集成

平台采用 **PAAP Server + PAAP Operator** 双层架构，详细设计参见 [Operator 设计方案](operator-design.md)。

```
┌─────────────────────────────────────────────────┐
│                   PAAP 前端                      │
└─────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────┐
│                PAAP Server（业务层）              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐       │
│  │ 应用服务 │ │ 环境服务 │ │ 权限服务 │       │
│  └──────────┘ └──────────┘ └──────────┘       │
│  ┌──────────┐ ┌──────────┐                     │
│  │ 模板渲染 │ │ CR 管理  │                     │
│  └──────────┘ └──────────┘                     │
└──────────────────────┬──────────────────────────┘
                       │ 创建/更新 CR
                       ▼
┌─────────────────────────────────────────────────┐
│              PAAP Operator（控制层）              │
│  ┌──────────────┐ ┌──────────────┐             │
│  │ Environment  │ │ ServiceInst  │             │
│  │ Controller   │ │ Controller   │             │
│  └──────────────┘ └──────────────┘             │
│  ┌──────────────┐                               │
│  │ Component    │                               │
│  │ Controller   │                               │
│  └──────────────┘                               │
└──────────────────────┬──────────────────────────┘
                       │ 管理 K8s 资源
                       ▼
┌─────────────────────────────────────────────────┐
│                 K8s 集群                         │
│  Namespace × N  │  SA/Role/RoleBinding          │
│  Deployment     │  Service/ConfigMap             │
│  NetworkPolicy  │  ResourceQuota                 │
└─────────────────────────────────────────────────┘
```

**职责划分：**
- **PAAP Server**：业务逻辑、API、模板渲染、CR 创建、状态同步
- **PAAP Operator**：K8s 资源管理、状态汇报、漂移修复、级联删除

## 十一、实施路线

### Phase 1：基础框架（4周）
- 用户认证与权限
- 应用CRUD
- 环境管理（模板创建）
- 基础导航与布局

### Phase 2：Operator + 核心服务（8周）
- PAAP Operator 开发（CRD 定义 + Controller）

- Component Controller（Deployment, Service 管理）
- 部署服务（ArgoCD 集成，通过 ServiceInstance CR 管理）

### Phase 3：监控与告警（4周）
- 监控服务（Prometheus 集成，通过 ServiceInstance CR 管理）
- 告警规则配置
- 通知渠道集成

### Phase 4：高级功能（4周）
- 自动化流程（PR触发部署）
- 高级模板编辑
- 操作审计日志
- 性能优化
